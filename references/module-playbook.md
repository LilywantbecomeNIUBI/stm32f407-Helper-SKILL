# 模块执行模板

用统一方式处理高频 STM32F407 电赛任务。先读当前工程，再读本地例程，再定位教材，再判断是否需要先回到 CubeMX。

每个模块默认都按下面 4 件事输出：

1. 该先看哪个例程和教材章节
2. 需要用户在 CubeMX 确认什么
3. 代码落到哪些文件
4. 最小联调步骤是什么

## GPIO / KEY / BEEP / EXTI

- 先看：
  GPIO、KEY、BEEP、EXTI 对应例程；教材第 13 到 16 章。
- 先确认的 CubeMX 项：
  GPIO 模式、上下拉、输出默认电平、EXTI 边沿、NVIC。
- 推荐落点：
  `Core/Inc/bsp_key.h`、`Core/Src/bsp_key.c` 或同类模块文件；主流程放 `main.c`；中断逻辑优先放 `HAL_GPIO_EXTI_Callback()`。
- 最小联调：
  先验证引脚电平是否变化，再验证回调是否进入，再验证业务逻辑是否触发。

## USART

- 先看：
  串口通信例程；教材第 17 章。
- 先确认的 CubeMX 项：
  USART 实例、波特率、TX/RX 引脚、NVIC、是否启用 DMA。
- 推荐落点：
  协议或上层逻辑放 `app_uart.c/.h`；保留 CubeMX 生成的初始化；`printf` 重定向或接收状态机按工程现状放 `main.c` 或独立模块。
- 最小联调：
  先确认串口工具参数一致，再看发送是否正常，再看接收回调或 DMA 路径是否进入。

## TIM / PWM / 输入捕获 / 编码器

- 先看：
  基本定时器、通用定时器、高级定时器例程；教材第 20 到 22 章。
- 先确认的 CubeMX 项：
  TIM 实例、Channel、工作模式、Prescaler、Period、GPIO AF、NVIC、DMA。
- 推荐落点：
  上层控制逻辑放 `app_tim.c/.h` 或模块文件；占空比设置、测速、捕获解析放独立函数；中断优先走 HAL 回调。
- 最小联调：
  先确认频率和占空比是否正确，再确认计数或捕获值是否变化，最后验证业务逻辑。

## ADC / DMA

- 先看：
  DMA、ADC、ADC + DMA 例程；教材第 30 到 33 章。
- 先确认的 CubeMX 项：
  ADC 实例、通道、Rank、采样时间、连续模式、DMA 绑定、触发源。
- 推荐落点：
  `app_adc.c/.h` 或独立采样模块；缓冲区与滤波逻辑放模块内部；结果显示或上传交给上层。
- 最小联调：
  先看缓冲区是否被刷新，再看采样值是否随输入变化，最后看单位换算和滤波结果。

## I2C

- 先看：
  IIC 例程；教材第 36 章；目标器件的寄存器表或通信时序。
- 先确认的 CubeMX 项：
  I2C 实例、SCL/SDA 引脚、速率、上拉、是否开中断或 DMA。
- 推荐落点：
  设备驱动放 `bsp_xxx.c/.h`；保留 HAL I2C 句柄；把寄存器读写和设备上层接口分开。
- 最小联调：
  先确认总线空闲和地址正确，再看 HAL 返回值，再看设备 ID 或寄存器读写是否正确。

## SPI

- 先看：
  SPI 例程；教材第 37 章；目标器件时序。
- 先确认的 CubeMX 项：
  SPI 实例、主从模式、CPOL/CPHA、NSS 管理、分频、GPIO AF、是否用 DMA。
- 推荐落点：
  设备驱动放 `bsp_xxx.c/.h`；CS/DC/RES 等控制脚与设备驱动同模块管理。
- 最小联调：
  先验证片选时序，再验证寄存器读写或设备 ID，最后验证批量收发。

## OLED / TFT

- 先看：
  OLED 或 TFT 对应例程；教材第 24、25 章。
- 先确认的 CubeMX 项：
  底层总线是 I2C、SPI 还是 FSMC；控制脚；DMA；时钟与总线速率；供电与背光。
- 推荐落点：
  底层驱动放 `bsp_oled.c/.h` 或 `bsp_lcd.c/.h`；上层显示逻辑放 `app_display.c/.h`；先做最小显示，再扩展图形接口。
- 最小联调：
  先确认供电和背光，再确认初始化命令发出，再确认是否能显示固定字符串。

## 常见传感器

- 先看：
  对应传感器例程；教材第 41 到 45 章，以及其底层总线章节。
- 先确认的 CubeMX 项：
  依赖的 GPIO、I2C、SPI、ADC、TIM、EXTI 以及板载资源冲突。
- 推荐落点：
  传感器驱动放 `bsp_sensor_xxx.c/.h`；若有多传感器融合，再单独建 `app_sensor.c/.h`。
- 最小联调：
  先读设备 ID 或原始数据，再做单位换算，最后再接业务逻辑。

## 模块迁移与裁剪规则

从教材或例程迁移模块时，按这个顺序裁剪：

1. 先保留最小可运行驱动接口。
2. 再保留必要宏、状态变量和时序处理。
3. 再保留与当前任务直接相关的显示、通信或采样流程。
4. 删除演示 UI、无关测试逻辑、与当前任务无关的封装层。
5. 删除与 Keil/MDK 工程结构强绑定的外壳文件。
6. 改造成可以直接接入 `Core/Inc + Core/Src` 的形式。

## 多文件补丁规则

当任务需要修改现有工程时，优先输出：

1. 新增哪些文件
2. 修改哪些文件
3. 每个文件承担什么职责
4. 修改顺序
5. 是否依赖 CubeMX 重新生成

不要只给一段孤立代码而不说明落点。

## 排错优先级

当用户反馈“没反应”“现象不对”“不工作”时，按这个顺序排查：

1. `.ioc` 是否与需求一致
2. 外设和时钟是否真正启用
3. GPIO / AF / 通道映射是否正确
4. DMA / NVIC 是否打开
5. `MX_XXX_Init()` 是否被调用
6. 用户代码是否真的执行到
7. 回调或中断是否进入
8. 板载资源是否冲突
9. 迁移时是否漏了必要文件、宏或时序
